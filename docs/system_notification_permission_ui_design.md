# 系统通知权限提示UI设计方案

## 问题背景
当用户拒绝系统通知权限后，即使应用内开启了系统通知开关，实际也无法发送警报通知。需要在UI中明确提示用户这一状态。

## 设计目标
1. 在噪音警报组件中显示权限状态提示
2. 在全局设置的警报通知栏目中显示权限状态提示
3. 提供清晰的用户引导，帮助用户开启系统通知权限
4. 确保两个位置的提示信息一致且美观

## UI设计方案

### 1. 噪音警报组件 (AlertsContent.ets)

#### 当前状态：
```
[系统通知] [开关]
在通知栏显示噪音警报信息
```

#### 改进后状态：

**情况1：权限正常**
```
[系统通知] [开关-开启]
警报触发时，系统将发送通知
```

**情况2：权限被拒绝**
```
[系统通知] [开关-开启]
⚠️ 系统通知已关闭，请开启系统通知
[去设置] 按钮
```

### 2. 警报通知设置 (NotificationPreferences.ets)

#### 当前状态：
```
[警报通知] [开关]
警报触发时，系统将发送通知。
```

#### 改进后状态：

**情况1：权限正常**
```
[警报通知] [开关-开启]
警报触发时，系统将发送通知。
```

**情况2：权限被拒绝**
```
[警报通知] [开关-开启]
⚠️ 系统通知已关闭，请开启系统通知
[去设置] 按钮
```

## 视觉设计规范

### 颜色方案
- **正常状态**：绿色 (#00BC7E)
- **警告状态**：橙色 (#FF9500)
- **错误状态**：红色 (#FF3B30)
- **文字颜色**：主文字 (#000000)，副文字 (#8E8E93)

### 图标设计
- 警告图标：⚠️ 或自定义警告图标
- 设置图标：⚙️ 或自定义设置图标

### 布局规范
```
┌─────────────────────────────────┐
│ [图标] 系统通知        [开关]    │
│                                 │
│ 描述文字                         │
│                                 │
│ [警告图标] 警告信息              │
│ [按钮] 去设置                   │
└─────────────────────────────────┘
```

## 交互流程

### 权限检查流程
```mermaid
graph TD
    A[组件加载] --> B[检查系统通知权限]
    B --> C{权限是否开启?}
    C -->|是| D[显示正常状态]
    C -->|否| E[显示警告状态]
    E --> F[用户点击"去设置"]
    F --> G[跳转到系统设置页面]
    G --> H[用户开启权限]
    H --> I[返回应用]
    I --> B
```

### 状态管理
- **权限正常**：显示正常描述文字，隐藏警告区域
- **权限被拒绝**：显示警告信息，显示"去设置"按钮
- **开关状态变化**：实时更新显示状态

## 技术实现要点

### 1. 权限状态检查
- 使用 `AlertService.checkNotificationPermission()` 方法
- 实时监听权限状态变化
- 缓存权限状态避免频繁检查

### 2. UI组件更新
- 在 `NotificationPreferences` 组件中添加权限状态显示
- 在 `AlertsContent` 组件中添加权限状态显示
- 使用条件渲染显示/隐藏警告区域

### 3. 跳转系统设置
- 使用系统API跳转到通知设置页面
- 提供友好的用户引导

## 预期效果

### 用户场景1：权限正常
用户看到清晰的提示："警报触发时，系统将发送通知"，知道系统通知功能正常工作。

### 用户场景2：权限被拒绝
用户看到醒目的警告："系统通知已关闭，请开启系统通知"，并有明确的"去设置"按钮，可以快速跳转到系统设置开启权限。

### 用户场景3：权限恢复
当用户在系统设置中开启权限后返回应用，界面自动更新为正常状态，用户获得即时反馈。

## 优势
1. **清晰明确**：用户一眼就能看出权限状态
2. **操作便捷**：提供直接的解决方案（去设置按钮）
3. **一致性**：两个位置的提示信息保持一致
4. **用户体验友好**：避免用户困惑，减少支持请求

这个设计方案将有效解决系统通知权限被拒绝时的用户体验问题。
